# Claw 知识库管理规则

## 📁 目录结构规范

### 1. 知识任务文件夹命名
- 每个知识任务必须使用**单独的文件夹**进行管理
- 文件夹命名必须使用**小写字母**和**中划线分割**
- 命名应简洁、清晰地描述知识任务的主题

**正确示例：**
```
ai-application-best-practices/
web-development-tutorials/
python-coding-standards/
github-workflow-guide/
```

**错误示例：**
```
AI_Application_BestPractices/  # 包含大写字母和下划线
web development tutorials/     # 包含空格
PythonCodingStandards/        # 驼峰命名
```

### 2. 文件夹内容结构
每个知识任务文件夹应包含：
```
knowledge-task-name/
├── README.md          # 任务概述和目录
├── content/           # 主要内容文件
├── examples/          # 示例代码或案例（如适用）
├── references/        # 参考资料和链接
└── metadata.json      # 任务元数据（可选）
```

## 🌿 分支管理规范

### 1. 开发流程
- **每个知识任务都必须在独立的分支上进行开发**
- 不允许直接在 `main` 分支上进行修改
- 分支命名应与知识任务文件夹名称保持一致

**分支命名示例：**
```
feature/ai-application-best-practices
feature/web-development-tutorials  
feature/python-coding-standards
```

### 2. 提交规范
- 提交信息应清晰描述所做的更改
- 每个提交应专注于单一功能或修复
- 避免将多个不相关的更改合并到一个提交中

### 3. 合并流程
- 所有更改必须经过**评审通过后**才能合入 `main` 分支
- 评审通过标准：
  - 内容准确性和完整性
  - 格式规范性
  - 是否符合知识库整体风格
- 合并前必须确保所有更改都在正确的知识任务文件夹中

## 📝 文档规范

### 1. Markdown 格式
- 使用标准 Markdown 语法
- 标题层级应合理（# → ## → ###）
- 代码块必须指定语言类型
- 链接和图片应使用相对路径（如适用）

### 2. 内容质量
- 内容应具有实用价值和可操作性
- 避免过于简略或冗长
- 重要概念应提供示例或说明
- 保持语言简洁明了

## 🔧 技术规范

### 1. 文件编码
- 所有文本文件必须使用 UTF-8 编码
- 行结束符使用 LF（Unix 风格）

### 2. 版本控制
- 避免提交二进制大文件
- 敏感信息不得提交到仓库
- 定期清理不必要的文件

## 🔄 维护规范

### 1. 定期更新
- 过时的内容应及时更新或标记为已过时
- 新的知识任务应遵循现有规范
- 定期审查和优化知识库结构

### 2. 问题处理
- 发现问题或需要改进的地方，应创建新的知识任务分支进行修复
- 重大结构调整需提前讨论和规划
